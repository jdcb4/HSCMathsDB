Executive Summary

The most reliable workflow has been a semi-automated import pipeline: cache official PDFs, extract rough text and question candidates, render the source pages to images, manually normalize each question into structured JSON, crop diagrams from rendered pages, then validate everything through schema checks, tests, build checks, source audits, and browser smoke tests.

The slowest part is not typing JSON. It is source verification: reading noisy PDF extraction, comparing against rendered pages and marking guides, mapping each question to syllabus nodes, and iterating diagram crops until they are both complete and clean.

The corrected ingestion standard is stricter than the early draft workflow: exam prompts should be near-verbatim, official marking-guide working should preserve the guide's steps, HSC marking feedback should be ingested for each available Section II question, and LLM worked solutions should live as an additional explanatory layer rather than replacing the raw/formatted official material.

Methodology Used

Catalogue official source packs

Each year is represented in sourcePacks with the NSW pack page, expected question count, course name, source collection, and import status. The source audit scripts compare the local catalogue to the currently visible NSW listings.

Download and cache PDFs

Exam papers, marking guidelines, and feedback PDFs are downloaded into ignored var/source-assets folders. These files remain local source evidence and do not pollute the app bundle.

Extract rough text and question candidates

Scripts extract text into var/extracted-text and candidate records into var/question-candidates. This speeds orientation but is treated as untrusted because mathematical notation, option layout, and page boundaries can be corrupted by PDF extraction.

Render PDF pages as source-of-truth images

Exam pages are rendered into var/rendered-pages. For diagrams, multiple choice options, graph labels, and poorly extracted mathematics, the rendered pages are more reliable than raw text.

Normalize records into the JSON corpus

Each question is manually promoted into src/data/hsc-math-advanced.json with near-verbatim TeX prompt text, official marking-guide answer and working steps, tags, source page references, and one or more syllabus node IDs. Records remain draft until independent review.

Ingest HSC marking feedback

Extracted marking-feedback text is parsed into per-question cards with better-response and improvement-area bullets. Multipart feedback is grouped back under part headings so users can see what belongs to each subquestion.

Crop and publish visual assets

Diagrams are cropped from rendered pages into ignored review folders, visually checked, then copied to public/assets/diagrams. Corpus records reference those public paths so the web UI can render them next to the question text.

Verify through code, data, source, and browser checks

The usual gate is data:validate, format, test, verify, data:audit-sources, data:audit-assets, coverage reporting, and a browser smoke test for a newly imported visual question.

Most Expensive Step

The dominant cost is resolving ambiguity from the source PDFs. Multiple-choice pages often extract as scrambled text, graph options are not reconstructable from text, and some marks or equations need checking against the marking guide. This forces repeated visual review.

Diagram work is the second largest cost because each crop needs several judgments: include all labels, remove irrelevant prompt fragments, avoid clipping axes or option curves, preserve enough context, and verify that the final public image actually loads in the app.

Options Considered

Use raw PDF text as the primary import source

Rejected as the primary path. It is useful for navigation, but it often corrupts symbols, fractions, option tables, and graph-heavy questions. It would create a high risk of quiet transcription errors.

Rejected

Recreate every graph as SVG or code

Rejected for initial corpus import. Code-native diagrams could be cleaner long term, but recreating official graph options by hand is slower and introduces fidelity risk. Exam-derived crops are faster and preserve the official visual context.

Rejected

Store everything directly in app components

Rejected. Structured JSON keeps the corpus queryable by year, topic, style, syllabus, and source pack. It also keeps future migration to SQLite or Postgres straightforward.

Rejected

Use a local JSON corpus first

Kept. For the current data size, JSON plus Zod validation is fast, simple, readable, and easy to review in git. A relational database can wait until editing workflows, concurrent writes, or richer query performance demand it.

Kept

Use subagents for parallel extraction review

Considered and attempted for splitting Section I review, but the session hit usage limits. The idea is sound for future runs: assign non-overlapping question ranges to cheaper read-only agents, then have the main agent integrate and verify.

Later

Batch image extraction from embedded PDF objects

Partially useful but not sufficient. Many exam diagrams are vector page content, not embedded bitmap images. Embedded extraction finds some assets, but rendered-page cropping remains necessary for vectors, labels, option groups, and shaded regions.

Partial

Bulk Image Extraction: Faster Path

Yes, there is a faster and better way than fully manual iteration, but it should be a semi-automated review queue rather than a blind bulk extractor. The hard part is not producing image crops. The hard part is knowing which crop is the complete official visual for a question.

1. Detect visual regions automatically

Use rendered page images plus computer vision heuristics to find non-text clusters: axes, curves, tables, shaded regions, and option grids. Output candidate bounding boxes per page.

2. Group boxes by question

Use detected question-number positions and page coordinates to associate visual regions with Q1-Q32. This reduces manual searching through pages.

3. Generate crop candidates in bulk

Write all candidate crops and JSON sidecars to var/diagram-crops, including source page, bounding box, confidence, and suspected question number.

4. Review in a contact sheet

Create an HTML contact sheet showing each candidate beside its question number and controls/status fields. A human or agent then approves, adjusts, or rejects crops.

Recommended Image Pipeline

  • Keep rendering each PDF page to high-resolution PNG as the canonical visual source.
  • Add a script such as data:detect-diagram-crops to propose bounding boxes.
  • Add data:build-crop-review to generate a local review HTML contact sheet.
  • Add a JSON manifest for accepted crops with stable IDs, source pages, and bounding boxes.
  • Add data:publish-reviewed-crops to copy accepted crops into public assets.
  • Continue doing a visual review before records reference the published assets.

Why Not Fully Automatic?

  • Some diagrams include surrounding prompt text that should be excluded.
  • Some multiple-choice questions need all option graphs in one image, not separate crops.
  • Tables may be better represented as TeX arrays instead of images.
  • Blank answer grids are sometimes unnecessary and should not be stored as assets.
  • Vector diagrams and labels can be visually connected even when object extraction splits them.

Better Use of Agents and Models

Task Best executor Reason
Source download, text extraction, page rendering Scripts Deterministic, cheap, repeatable, and easy to validate from file output.
Question prompt extraction by range Cheap read-only subagent Can run in parallel by non-overlapping question ranges and produce draft notes.
Answer checking and syllabus mapping Main agent or stronger reviewer Requires judgment and consistency with existing corpus conventions.
Diagram candidate generation Script plus vision review Bulk detection is mechanical, but crop acceptance needs visual judgment.
Final integration Main agent Prevents conflicting writes and keeps source counts, tests, docs, and browser checks aligned.

Practical Next Improvements

  • Build a crop-review contact sheet so the next year does not require repeated individual image opens.
  • Add source-page snapshots beside raw candidate text in the candidate report for faster prompt verification.
  • Use two or three read-only agents for independent extraction notes when quota permits, split by question ranges.
  • Add a small syllabus-mapping checklist per question style to reduce repeated reasoning.
  • Keep the current final gate: schema validation, coverage, tests, build, source audits, and one browser smoke check for a newly imported visual question.

This report is based on the workflow used in the current repository state after completing the 2022 Mathematics Advanced draft import. It describes the import process as practiced, not a theoretical ideal pipeline.
